# ReviewPilot 赛题对齐说明

## 1. 赛题原文

题目三：AI PR Review 助手

请开发一个 AI 代码评审工具，帮助开发者提升 Pull Request 的 Review 效率与质量。

要求：请了解开发者在代码评审中的真实需求，设计并实现一个以 AI 辅助分析为核心的工具。用户指定 GitHub PR，系统自动获取代码变更并智能分析，辅助发现问题。需支持 PR 变更总结、风险代码识别、Review 建议生成等。请综合考虑分析准确性、上下文理解、误报与漏报控制、响应速度及使用体验等关键因素，并在作品中说明系统在模型选择、上下文获取方式及未来扩展方向上的设计思路。

## 2. 项目定位结论

ReviewPilot 的方向与题目三直接对应。它不是普通聊天机器人，也不是只做静态扫描的规则工具，而是一个围绕 Pull Request 变更展开的 AI Review 工作台。

一句话定位：

> ReviewPilot 是一个基于 GitHub PR diff 和代码证据链的 AI 代码评审助手，用千问生成中文 PR 摘要、风险识别和 Review 建议。

## 3. 与赛题要求的对应关系

| 赛题要求 | ReviewPilot 对应设计 | 展示重点 |
| --- | --- | --- |
| 用户指定 GitHub PR | 输入公开 GitHub PR 链接 | 无需 GitHub Token，适合公开仓库演示 |
| 自动获取代码变更 | 调用 GitHub 公开 PR 接口并下载 diff | 能从 PR 链接进入分析流程 |
| 智能分析 | 构建证据链后调用千问 | AI 不是自由发挥，而是基于证据分析 |
| PR 变更总结 | 返回 summary 字段 | 快速理解本次 PR 改了什么 |
| 风险代码识别 | findings 包含 file、line、severity、evidence | 每条风险绑定具体文件和证据 |
| Review 建议生成 | findings 中包含 suggestion | 直接给出可执行修改建议 |
| 分析准确性 | 后端过滤无效文件和无证据 finding | 降低模型幻觉 |
| 上下文理解 | 传入 PR 标题、描述、变更文件和新增代码证据 | 让模型理解 PR 背景和代码变化 |
| 误报与漏报控制 | 区分 severity 和 confidence | 不把不确定判断伪装成确定问题 |
| 响应速度 | MVP 限制 evidence 数量 | 避免一次传入过大 diff |
| 使用体验 | 单页工作台，支持 PR 链接和粘贴 diff | 演示稳定，操作路径短 |
| 模型选择说明 | 默认使用千问 qwen-plus | 适合中文报告和国内 API 场景 |
| 未来扩展方向 | 分块分析、代码图谱、规则配置、模拟评论 | 体现系统可继续演进 |

## 4. 为什么不只做“把 diff 发给大模型”

直接把完整 diff 交给模型有三个问题：

- 上下文过长时容易遗漏重点。
- 模型可能编造不存在的文件、函数或风险。
- 输出容易模板化，例如反复说“建议增加测试”。

ReviewPilot 的应对方式：

- 先解析 diff，再抽取 changedFiles 和 evidence。
- Prompt 明确要求只根据证据分析。
- 后端过滤未引用变更文件或无证据的 finding。
- 输出结构化 JSON，前端按字段展示。

## 5. 模型选择说明

当前版本默认使用千问：

- 中文能力较好，适合中文项目说明和答辩展示。
- 支持 OpenAI 兼容调用方式，后续可替换其他模型 Provider。
- API Key 保存在本地环境变量，不进入前端和 Git 仓库。

后续可扩展：

- 增加 DeepSeek、OpenAI 或本地模型 Provider。
- 按 PR 类型选择不同模型，例如小 PR 用快速模型，大 PR 用更强推理模型。

## 6. 上下文获取方式

ReviewPilot 当前上下文来源包括：

- PR 标题
- PR 描述
- 变更文件列表
- 每个文件新增/删除行数
- hunk 信息
- 新增代码证据
- 文件类别识别结果

后续计划补充：

- 同文件未修改上下文片段
- import 关系
- 函数调用关系
- 测试文件覆盖关系
- 仓库级 Review 规则

## 7. 误报与漏报控制

降低误报：

- 每条 finding 必须引用本次变更文件。
- 每条 finding 必须有 evidence。
- 低置信度建议不作为高风险问题展示。

降低漏报：

- 对认证、支付、数据、配置、前端、测试等文件分类。
- 对不同类别文件给模型不同关注重点。
- 后续加入轻量代码图谱，补充跨文件上下文。

## 8. Demo 展示建议

Demo 不要只说“我调用了一个大模型”。建议按下面顺序展示：

1. 输入公开 GitHub PR 或加载样例 diff。
2. 展示系统解析出的变更文件。
3. 展示证据数量和风险等级。
4. 展示一条绑定文件与证据的 Review 建议。
5. 说明后端如何过滤无证据输出。
6. 打开文档说明未来扩展方向。

## 9. 最终建议

后续开发优先围绕这条主线推进：

```text
PR 获取
-> diff 解析
-> 证据链
-> 千问 Review
-> 结果校验
-> 报告展示
```

不要过早扩展成通用代码聊天工具，也不要把重点放在复杂账号系统上。评审最容易认可的是：这个工具确实理解 PR Review 流程，并对 AI 幻觉和证据约束做了工程设计。
