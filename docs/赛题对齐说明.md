# ReviewPilot 赛题对齐说明

## 1. 赛题定位

赛题要求实现一个 AI PR Review 助手：用户指定 GitHub PR 后，系统自动获取代码变更并智能分析，辅助发现问题，输出 PR 变更总结、风险代码识别和 Review 建议，同时需要考虑分析准确性、上下文理解、误报漏报控制、响应速度和使用体验。

ReviewPilot 当前版本已经围绕这一目标形成完整闭环：从 PR 链接 / unified diff / `.diff` 文件输入，到风险感知分析、规则预检、Reviewer-Auditor 双模型审计、代码评审工作台、Ask PR 追问、历史记录和 Markdown 报告导出。

## 2. 与赛题要求的对应关系

| 赛题要求 | ReviewPilot 当前实现 |
| --- | --- |
| 指定 GitHub PR | 支持输入公开 GitHub PR 链接并获取 diff |
| 自动获取代码变更 | 解析 PR diff、文件状态、新增/删除行、hunk 和 file_diffs |
| PR 变更总结 | 输出总览、核心指标、变更模块和重点文件摘要 |
| 风险代码识别 | 根据文件风险分、规则预检和 AI 分析生成风险列表 |
| Review 建议生成 | 每条问题可复制 Review Comment，并支持 Ask PR 追问 |
| 上下文理解 | 使用 PR 信息、文件关系、变更类型、规则命中和风险信号构造上下文 |
| 误报漏报控制 | Reviewer 初审后由 Auditor 复核，最终融合高可信风险、潜在风险和需人工复核项 |
| 响应速度 | 快速分析与深度审计分离，用户可按场景选择 |
| 使用体验 | 深色工作台、风险文件列表、Unified Diff、AI 问题卡片、历史记录和报告导出 |
| 模型选择 | 支持 OpenAI-Compatible 模型配置，可接入 Qwen、DeepSeek、OpenAI 等服务 |

## 3. 核心实现亮点

- **风险感知文件排序**：根据文件类型、变更规模、路径关键词、文件状态和潜在影响范围计算优先级，让 Reviewer 先看高风险文件。
- **规则预检**：在大模型分析前先进行确定性规则扫描，用于发现异步错误处理、路由兜底、Context/Provider、配置和依赖变更等候选问题。
- **Reviewer-Auditor 双模型审计**：Reviewer 负责初审，Auditor 负责复核证据、误报、漏报和置信度，降低单模型输出的不确定性。
- **代码评审工作台**：左侧风险文件列表，中间 Unified Diff，右侧 AI 问题卡片，形成接近真实 PR Review 的阅读路径。
- **Ask PR 交互追问**：用户可以围绕当前报告继续问问题，聊天区支持即时显示用户消息、AI 生成状态、失败重试和快捷问题。
- **历史记录与报告导出**：保存过往 Review 结果，支持复盘和导出 Markdown 报告。

## 4. 算法亮点：轻量化 Review Knowledge Graph

ReviewPilot 不是直接把完整 diff 丢给大模型，而是先把 PR 信息整理成一张轻量化 Review Knowledge Graph / Review 关系图，再引导 Reviewer 与 Auditor 推理。

这张图谱不是大型静态分析系统，而是服务于 PR Review 的轻量结构化上下文。它把以下信息组织到同一套关系中：

- PR 概览、标题和描述
- 变更文件、文件类型、文件状态和新增/删除规模
- 风险评分、规则预检命中和上下文覆盖情况
- 关键路径文件，例如入口、路由、状态管理、认证、支付、配置和依赖文件
- 测试相关线索，例如业务文件变化但测试文件未变化
- Reviewer 初审风险、Auditor 复核结论和最终融合结果

这种设计参考了 KnowGPT 中“先构建知识结构，再引导大模型推理”的思想。相比单纯调用大模型 API，ReviewPilot 先提取文件关系、变更类型、风险信号和上下文线索，再让模型基于结构化关系进行审查，有助于提升可解释性，并减少误报和漏报。

## 5. 为什么不是简单的大模型套壳

直接把完整 diff 发给模型容易出现三个问题：

- 大 PR 上下文过长，模型容易遗漏重点。
- 模型可能编造不存在的文件、函数、接口或行号。
- 输出容易模板化，只给泛泛的“建议补充测试”。

ReviewPilot 的处理方式是：

- 先解析 diff 和文件结构，再选择高价值上下文。
- 在模型前增加规则预检和风险排序。
- 用 Reviewer-Auditor 两阶段审计控制误报漏报。
- 要求问题绑定文件、位置、证据和建议。
- 前端以问题卡片和 Review 操作呈现，而不是只展示长篇聊天文本。

## 6. 当前可展示的完成度

当前版本适合按以下路径进行 Demo：

1. 输入 GitHub PR 链接、粘贴 unified diff 或上传 `.diff` / `.patch` 文件。
2. 查看总览页，理解整体风险、重点文件和测试建议。
3. 进入代码评审页，按风险文件列表选择文件。
4. 在 Unified Diff 中查看代码变更和行内 AI 标记。
5. 在右侧 AI 问题卡片中展开详情、复制 Review Comment 或 Ask PR 追问。
6. 在 Ask PR 中继续询问高风险文件、潜在 Bug、测试覆盖不足和人工复核重点。
7. 在历史记录中查看过往报告并导出 Markdown。

## 7. 当前限制与后续优化

- 当前主要面向本地运行和演示场景，在线 Demo 地址仍需部署。
- 免费模型 API 可能存在延迟、限流或超时，需要准备备用模型配置。
- 轻量化知识图谱目前服务于 Review 上下文组织，后续可进一步增强跨文件 import、调用链和测试影响分析。
- 大型 PR 的分块分析、增量缓存和长上下文压缩仍可继续优化。
- 后续可接入 GitHub Review Comment API，实现从 AI 建议到 PR 评论的闭环。
