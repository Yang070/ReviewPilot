# ReviewPilot Demo 讲解稿

## 1. Demo 目标

本 Demo 展示 ReviewPilot 如何把一个 GitHub PR 或 diff，转化成可执行的 AI 代码评审工作台。演示重点不是“让大模型泛泛总结代码”，而是围绕 PR 变更、风险文件、Unified Diff、AI 问题卡片、Ask PR 追问和知识图谱证据链，完成一次接近真实 Code Review 的流程。

核心亮点：

- 风险优先：先看到最值得人工复核的文件和问题。
- 证据绑定：AI 建议绑定到文件、代码行和 diff 证据。
- Unified Diff 阅读：代码区只保留统一视图，减少 Split 布局带来的干扰。
- Ask PR 追问：用户提问后立即显示消息和生成状态，返回后替换为真实回答。
- 知识图谱创新：用轻量代码图谱理解文件关系、依赖关系和变更影响范围。

## 2. Demo 前准备

启动后端服务：

```powershell
python server.py
```

打开本地页面：

```text
http://127.0.0.1:8770
```

演示前确认：

- 已登录 ReviewPilot。
- 模型设置页已配置可用模型。
- 准备一个 GitHub PR 链接，或准备一段 diff / patch 文本。
- 选择“深度审计”时，可展示 Reviewer 与 Auditor 的交叉复核效果。

## 3. 推荐视频结构

建议时长：4 到 6 分钟。

### 3.1 开场：说明项目解决的问题

讲解词：

```text
大家好，我演示的是 ReviewPilot，一个面向 Pull Request 的 AI 代码评审助手。

在真实开发里，一个 PR 往往包含多个文件、不同风险等级和大量上下文。人工 Review 的难点不是看不懂某一行代码，而是很难快速判断：哪些文件最危险、哪些问题必须人工复核、哪些建议可以直接复制成 Review Comment。

ReviewPilot 的目标就是把 PR 变成一个 AI Review 工作台，让开发者按“风险文件列表、Unified Diff、AI 问题卡片、Ask PR 追问”的路径完成评审。
```

画面操作：

- 打开 ReviewPilot 首页。
- 简单展示 PR URL、粘贴 diff、上传 patch 三种输入方式。
- 进入模型配置选择区，但不要展开过多设置。

### 3.2 输入 PR 或 diff：启动评审

讲解词：

```text
这里我可以输入 GitHub PR 链接，也可以直接粘贴 unified diff。为了演示稳定，我使用当前准备好的 PR / diff。

点击开始评审后，系统会先解析变更文件，提取每个文件的新增删除行、变更类型和风险线索；如果选择深度审计，还会先由 Reviewer 给出初审，再由 Auditor 做复核，最后融合成最终报告。
```

画面操作：

- 粘贴 PR 链接或 diff。
- 选择模型和分析模式。
- 点击开始评审。
- 展示分析进度。

### 3.3 总览页：先看风险分布

讲解词：

```text
评审完成后，首先看总览。这里不是简单给一个大段总结，而是把 PR 拆成几个核心维度：整体风险、关键文件、问题数量、测试建议和需要人工确认的点。

这样 Reviewer 第一眼就能知道这次 PR 是偏低风险的文案修改，还是涉及状态流转、接口调用、权限、安全或测试缺口的高风险变更。
```

画面操作：

- 展示概览卡片。
- 指向高风险文件、问题数、测试建议。
- 简短停留，不展开所有细节。

### 3.4 代码评审页：风险文件列表到 Unified Diff

讲解词：

```text
接下来进入代码评审页。左侧是风险文件列表，按照高风险、有 AI 问题、需要人工复核这些条件排序。

我点击一个文件后，中间是 Unified Diff 主阅读区。这里我保留统一 diff 视图，不再使用 Split Diff，因为当前项目的核心目标是让 Review 路径更稳定、更清楚：先看变更，再看对应的 AI 问题。

顶部只保留文件名、风险等级、新增删除行数、AI 问题数量和简短风险原因。Ask PR 快捷问题和技术细节都默认折叠，避免占用代码阅读空间。
```

画面操作：

- 点击左侧高风险或中风险文件。
- 展示 Unified Diff。
- 展示新增/删除代码高亮、行号、AI 批注标记。
- 展开一次“技术细节”，再收起。

### 3.5 AI 问题卡片：从建议到 Review 操作

讲解词：

```text
右侧是 AI 问题面板。每条问题以卡片形式展示，包括问题类型、风险等级、文件位置和是否需要人工复核。

问题默认不展示长篇解释，只有点击展开详情时，才会看到为什么重要、建议怎么改、证据代码是什么。这样页面不会被 AI 文本淹没，Reviewer 可以先处理高优先级问题。

每条问题都提供 Review 操作，例如复制 Review Comment、Ask PR 追问。如果只是当前页面判断不需要处理，也可以用前端状态临时忽略或标记。
```

画面操作：

- 展示右侧问题卡片。
- 展开一条高风险问题。
- 点击复制 Review Comment。
- 点击 Ask PR 追问入口。

### 3.6 Ask PR：标准聊天式追问

讲解词：

```text
ReviewPilot 还支持 Ask PR。这里不是重新分析整个 PR，而是围绕当前报告继续问问题。

比如我问：这个文件最需要人工复核的地方是什么？发送后，用户问题会立刻出现在聊天区，同时出现“正在分析当前 PR”的 AI 占位消息。等模型返回后，占位消息会被真实回答替换。

这个交互和常见聊天产品一致，用户不会误以为页面卡住；如果请求失败，原问题仍然保留，并提供重试入口。
```

画面操作：

- 输入一个问题。
- 展示用户消息立即出现。
- 展示 AI loading。
- 展示回答返回。
- 点击一个快捷问题，说明快捷问题也走同一套流程。

### 3.7 创新点：知识图谱辅助代码评审

讲解词：

```text
这里我重点介绍 ReviewPilot 的创新点：知识图谱辅助代码评审。

传统 AI Review 往往只看当前 diff，很容易忽略跨文件影响。ReviewPilot 在解析 PR 时，会构建轻量代码图谱，把文件、import 依赖、函数线索、测试文件和变更模块组织成关系网络。

这个图谱不追求做完整 IDE 级静态分析，而是服务于 Review 场景：它帮助系统回答三个问题。

第一，这个文件是不是关键路径文件，比如入口、路由、状态管理、认证、支付或配置。

第二，这个改动可能影响哪些相邻文件，比如某个 Context、Provider、工具函数或组件被多个地方依赖。

第三，当前变更有没有对应测试或验证路径。如果业务文件变了，但测试文件没有变化，系统会把它作为测试覆盖不足的线索。

所以 ReviewPilot 的 AI 建议不是孤立生成的，而是结合 diff 证据、规则预检、风险排序和知识图谱上下文来判断。这样可以减少空泛建议，让每条问题更接近真实 Reviewer 会关注的点。
```

画面操作：

- 回到概览或变更文件页面。
- 指出文件分类、关键路径、测试覆盖、依赖/配置变化等信息。
- 如果页面中有结构化分析或技术细节，展开并指出“结构化信息”和“风险原因”。

### 3.8 结尾：总结价值

讲解词：

```text
总结一下，ReviewPilot 做的不是替代人工 Reviewer，而是帮 Reviewer 更快进入状态。

它先用风险排序告诉我哪些文件优先看，再用 Unified Diff 保留代码证据，再用 AI 问题卡片沉淀 Review 建议，最后通过 Ask PR 支持继续追问。

其中知识图谱是我认为最有价值的创新点：它把代码文件之间的关系引入 AI Review，让模型不只看单行 diff，而是理解这次 PR 可能影响的上下文。

这就是 ReviewPilot 的核心演示，谢谢大家。
```

画面操作：

- 快速回放：输入 PR / 总览 / 代码评审 / Ask PR。
- 停在代码评审工作台或总览页结束。

## 4. Demo 讲解重点备忘

- 不要把页面当成普通聊天机器人介绍，要强调它是 PR Review 工作台。
- 代码评审页按“左侧风险文件列表 -> 中间 Unified Diff -> 右侧 AI 问题卡片 -> Ask PR 追问”讲。
- 知识图谱不要讲成复杂论文系统，要讲成“为 AI Review 提供跨文件上下文的轻量关系网络”。
- 演示时优先展示高风险、有问题、需人工复核的文件。
- 技术细节只展开一次，说明默认折叠是为了让 Reviewer 专注读代码。
