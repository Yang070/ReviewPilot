# ReviewPilot 需求文档

## 1. 项目概述

ReviewPilot 是一个面向开发者的 AI Pull Request 代码评审助手。用户输入公开 GitHub PR 链接，或直接粘贴 unified diff，系统会自动获取代码变更、抽取评审证据，并调用千问生成结构化 Review 报告。

本项目的目标不是替代人工 Review，而是帮助开发者在正式合并前更快理解 PR 内容，提前发现高风险代码、测试缺口和需要人工重点确认的位置。

## 2. 项目定位

- 项目名称：ReviewPilot
- 产品形态：本地 Web 工具
- 核心用户：学生开发者、开源项目维护者、小团队开发者、代码评审新人
- 核心场景：PR 合并前的辅助审查
- 核心价值：提升 Review 效率，降低漏看风险，并让 AI 建议具备可追溯证据

## 3. 背景与痛点

真实代码评审中常见问题包括：

- PR 文件多、diff 长，人工很难快速抓住重点。
- Review 新人不知道应该优先检查安全、权限、边界条件还是测试。
- 大模型直接评审容易出现幻觉，给出没有代码证据的建议。
- 传统静态规则能发现部分危险模式，但难以生成自然、可读的评审说明。
- 团队希望 PR 描述、实现思路、测试方式更规范，便于后续追溯。

ReviewPilot 通过“规则化证据抽取 + 大模型中文评审”的方式，把 AI 放进一个可约束、可校验的流程里。

## 4. 用户画像

### 4.1 学生开发者

- 目标：在课程项目或比赛中提交更规范的代码。
- 关注点：工具是否容易运行，报告是否能指出具体文件和建议。

### 4.2 开源维护者

- 目标：快速理解外部贡献者的 PR。
- 关注点：变更总结、风险定位、测试建议和误报控制。

### 4.3 小团队开发者

- 目标：减少低质量 PR 进入主分支。
- 关注点：公开仓库可直接分析，不需要复杂权限配置。

## 5. MVP 目标

当前版本优先完成：

- 输入公开 GitHub PR 链接并自动获取 diff。
- 支持粘贴 diff 作为兜底方案。
- 解析变更文件、hunk、新增行和删除行。
- 构建代码证据链。
- 调用千问生成中文结构化 Review 报告。
- 展示 PR 摘要、风险代码、Review 建议和测试建议。
- 对没有证据的 finding 做后端过滤。

## 6. 功能需求

### 6.1 PR 输入

用户可以输入公开 GitHub PR 链接。

验收标准：

- 支持 `https://github.com/{owner}/{repo}/pull/{number}` 格式。
- 私有仓库或不存在的 PR 给出明确错误提示。
- GitHub 频率限制时提示用户改用粘贴 diff。

### 6.2 Diff 粘贴分析

用户可以直接粘贴 unified diff。

验收标准：

- 没有 GitHub 网络时仍可演示。
- diff 解析失败时提示“未解析到变更文件”。

### 6.3 变更解析

系统解析变更文件、hunk、新增行、删除行和行号。

验收标准：

- 能统计每个文件的新增/删除行数。
- 能提取新增代码作为模型证据。
- 能识别认证、支付、数据、配置、前端、测试等文件类别。

### 6.4 AI Review

系统调用千问生成中文 Review 报告。

验收标准：

- 输出包含 summary、riskLevel、findings、testSuggestions。
- 每条 finding 必须包含 file、evidence、severity、message、suggestion、confidence。
- 输出统一为中文，便于项目展示和答辩。

### 6.5 幻觉控制

后端对模型输出进行校验。

验收标准：

- finding 引用的文件必须存在于本次变更文件中。
- finding 必须包含非空 evidence。
- 非法 severity 自动降级为 low。
- confidence 被限制在 0 到 1 之间。

## 7. 非功能需求

- 易用性：页面首屏即为工作台，不做营销页。
- 安全性：千问 API Key 只存在后端环境变量中，不进入前端代码和 Git 仓库。
- 响应速度：MVP 阶段优先限制证据数量，避免一次传入过大 diff。
- 可维护性：GitHub 获取、diff 解析、模型调用、Review 服务分模块实现。
- 可扩展性：后续可扩展文件分块、代码图谱、规则配置和 GitHub Review 评论。

## 8. 技术方案

- 后端：Python 标准库 HTTP 服务
- 前端：原生 HTML、CSS、JavaScript
- 模型：千问 `qwen-plus`
- GitHub：公开 PR API + diff URL
- 测试：Python unittest

## 9. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 千问 API Key 未配置 | 无法生成真实 AI 报告 | README 明确配置方式，错误提示清晰 |
| GitHub 访问失败 | 无法自动拉取 PR | 支持手动粘贴 diff |
| 模型幻觉 | 输出不可信建议 | 使用证据链和后端校验过滤 |
| 大 PR 内容过多 | 响应慢或超过上下文 | MVP 限制证据数量，后续加入分块 |
| 建议过于模板化 | 使用体验差 | 按文件类别和证据构造 prompt |

## 10. 验收标准

最终提交时应满足：

- README 能说明本地运行方式。
- 能完成一次完整 demo：输入 PR 或 diff，生成 Review 报告。
- 文档说明模型选择、上下文获取方式、误报与漏报控制、未来扩展方向。
- 有持续 commit 和 PR 记录。
- 不在代码中硬编码 API Key。
